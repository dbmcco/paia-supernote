## End State

The paia-supernote service has a durable SQLite-backed Cloud change ledger for allowlisted notebooks, exposes safe cached agent read/write contracts with cursor-based reads and base-revision guarded writes, and returns structured recoverable error guidance across existing CLI/event/API-like surfaces without weakening dry-run or upload safety.

## Acceptance Criteria

- Configuration supports an explicit allowlist of Cloud notebooks for ledger-backed ingest; non-allowlisted notebooks are ignored before download/OCR/mutation where existing Cloud metadata allows.
- Ledger snapshots store notebook revision identity, stable page IDs, ordered page lists, page content hashes, and notebook content hashes in SQLite or the existing configured state store.
- Diffing records added, updated, removed, and reordered pages with enough metadata for explicit-cursor and per-agent-cursor reads.
- OCR is invoked only for pages whose stable ID is new or whose content hash changed; reorder-only diffs do not trigger OCR.
- Cached reads return latest allowlisted notebook/page state and change batches without contacting Supernote Cloud.
- Write routes require caller-supplied base revision or ledger revision and fail closed on stale, missing, or mismatched revisions before upload/apply/S3 calls.
- CLI, event, and any in-repo API-like surfaces use Pydantic models with machine-readable code, message, retryability, next actions, and conflict details where applicable.
- Existing safe upload behavior remains intact: dry-run paths do not mutate Cloud data, real notebook writes remain explicitly guarded, and tests do not touch production Supernote notebooks.
- Focused unit tests, offline integration tests, ruff, git diff checks, roborev, and final pytest gates pass.

## Out Of Scope

- Real Supernote Cloud notebook mutations during test execution.
- New browser organizer UI features or UI redesign.
- Native Supernote cross-notebook links or generated index pages.
- Dependency upgrades or uv.lock churn unless strictly required and explicitly reviewed.
- Broad refactors unrelated to ledger, cached reads, write concurrency, and structured error propagation.
- Automatic Quick.note triage or page moving.

## Risks

- Stable page IDs may not exist in all parsed .note fixtures or older notebook formats. [high]: Implement and document a deterministic fallback hash strategy only after tests prove reorder stability; escalate if impossible.
- Base-revision validation against cached ledger state leaves a TOCTOU window if Cloud changes after the last poll and before write. [high]: Document the poll_interval-bounded freshness window in write-safety output and handoff; do not claim live Cloud conflict prevention unless approved design adds a live pre-write fetch.
- Base-revision checks might be bypassed by an older direct uploader call path. [high]: Use uploader spies and integration tests to prove conflict checks happen before upload and review direct upload APIs in roborev.
- Structured error envelopes may diverge across CLI, event, and API-like surfaces. [medium]: Centralize Pydantic models and test every reachable surface against the same serialized fields.
